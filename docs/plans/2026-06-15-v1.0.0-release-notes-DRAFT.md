Bifrost reaches 1.0 with **Solutions** — installable, deploy-owned, lifecycle-aware packages of apps, workflows, forms, agents, tables, and configs that can be shipped between environments, installed from a zip or a GitHub repo, developed locally, and updated in place. This release also ships a **new standalone-React app runtime (v2)**, a **magic-link user-invite + topic-event system**, an **external-user access tier**, a major **client UI overhaul**, and execution-sandbox hardening. It moves the project to the `gobifrost` GitHub org and container registry. 652 commits since v0.9.1.

## ⚠️ Release cadence going forward

Starting with this release, Bifrost ships on a three-rung ladder:

- **`:dev`** — built on every merge to `main`. **Bleeding edge; expect bugs.** This is where the maintainer flushes out defects in his own production before they reach anyone else.
- **`vX.Y.Z-rc.N` (pre-release)** — cut **roughly monthly, in between full releases**. Intended to be **safer than `:dev`**: a versioned, signed candidate that's been through the gates and is being soak-tested, but not yet blessed. Pre-releases do **not** move the `:latest` tag.
- **`vX.Y.Z` (full release)** — **roughly monthly**, blessed/stable, tagged `:latest`. The default for production installs.

If you want fresher-than-monthly without riding `:dev`, pin to the latest `-rc` tag.

## Features

- **Solutions** — package a set of platform entities (apps, workflows, forms, agents, tables, configs, event/schedule triggers) into an installable unit. Install from a workspace zip (`bifrost solution install <zip>`) or from a GitHub repo (including an omni-repo subpath); develop locally with `bifrost solution start` (one-origin, Firebase/SWA-style — app dev server + in-process workflows, deploys nothing); ship with `bifrost solution deploy`.
- **Read-only, deploy-owned installs** — a Solution's entities are read-only in the target environment (editing one live returns 409). Updates flow through `deploy`, or through auto-pull for git-connected installs. An always-on database guard backs this up.
- **Connection references + Setup wizard** — a Solution declares the integrations it needs; install surfaces unmet connection needs, blocks/flags them, and a Setup wizard + checklist + README tab guide the operator to completion.
- **New standalone-React app runtime (v2)** — apps are now true, self-contained React projects: each owns its own `createRoot` + router, mounts at `/apps/{slug}`, and imports the `bifrost` SDK as a real npm package (served from `/api/sdk/download`). New apps default to this `standalone_v2` model. The legacy `inline_v1` runtime (React/shadcn/router injected via `globalThis.__bifrost_platform`) remains supported for existing inline apps. v2 apps are scaffolded and shipped inside a Solution (`bifrost solution scaffold-app`).
- **Magic-link user invites + topic events** — invite users by email with single-use, expiring magic links; invitees register via passkey, password, or SSO on a branded `/accept-invite` page; the Users table shows invite status (Active / Pending / Expired / Not invited). A new **`bifrost.events` SDK module** (`await events.emit(topic, data, scope)`) plus `POST /api/events/emit` lets workflows publish and subscribe to topic events; `context.event` is now populated on all event-triggered workflows (#273).
- **Built-in platform events** — Bifrost now emits subscribable events for workflow failures, integration lifecycle/refresh, and delivery-retry exhaustion, with example payloads and in-UI event-source reference help (#321).
- **External users + the `everyone` access tier** — you can now mark a user as **external** (a portal/guest — e.g. *your customer's* customers). Previously every signed-in user counted as "authenticated," so an external user could reach anything set to the `authenticated` tier — including global apps you built for your own direct customers. Now `authenticated` means **"everyone except external users,"** and a new **`everyone`** tier explicitly opts an entity in for external users. External users still get access to anything granted by a role they're assigned, exactly as before. (#291 and the external-access work)
- **Custom Claims** — define org-scoped (or global) policy facts about the calling user (e.g. `allowed_campus_ids`) and reference them from any table policy as `{claims: <name>}`. Full management tab, CLI `--scope`, MCP `scope=` support (#291).
- **Bulk user actions + redesigned Roles** — multi-select bulk actions on the Users table (`PATCH /api/users/bulk`: move org / replace roles / activate-deactivate); role assignment now covers apps, workflows, and knowledge (previously only users/forms/agents) plus bulk-unassign; a redesigned `/roles` page with consumer-count chips that deep-link to each role's tabs (#277).
- **Per-entity logos for apps and agents** — upload a square logo per app/agent (PNG/JPEG/SVG, 5 MB), shown on a refreshed Apps grid; app delete moved into the settings danger zone (#269).
- **Per-mapping OAuth connections** — connect each org's integration mapping independently (token cascade mapping → org → global), with per-row Connect/Connected UI, expiry tooltips, inline refresh, and an `entity_id` source picker on first connect (#259).
- **Editable workflow tool names** — a workflow's `name` is explicitly its MCP tool name (defaults from the Python function), editable across REST/CLI/MCP/manifest with an optional display label — directly controls how agents and MCP clients see the tool (#346).
- **Configurable branding** — admins can set an application name and relabel app/agent/form terminology across the UI (#345, #379).

## Client UI overhaul

- The largest UI change in the release (#370): History rebuilt with day-grouping, status badges, skeleton/empty/error states, and a URL-synced multi-value `?status=` filter; the execution drawer/details rebuilt with coalesced tracebacks, copyable errors, and a new "Pretty View" output renderer; the Dashboard redone into four consolidated stat cards plus an executions-over-time chart; the Cmd+K palette rebuilt.

## Security

- **Execution sandbox hardening** — worker pods now run with a tightened security context (capability drop, seccomp, read-only root filesystem) and **credential-free execution children**, so workflow code can no longer inherit secrets that could mint tokens or decrypt integration data; the test stack was brought to the same posture (#363, #364).

## Reliability

- **Process-pool stability under burst load** — fixed race-prone dict mutation in worker-cleanup paths that crashed executions with "dictionary changed size" / KeyError and hung new work for 30s ("No worker slot available"), and fixed the on-demand pool's "No idle process available" timeout by waking slot waiters (#294, #319). *(An earlier completed-worker reaping change was rolled back this cycle after it caused a production crash storm — #293.)*
- **Resilient worker requirements install** — one unbuildable package no longer strips the entire workflow runtime (this took prod down on 2026-05-24); per-package fallback install, deduplicated admin notification, and fleet-aware install progress surfacing the real pip error in the UI (#299).
- **Worker/scheduler memory reductions** — import-closure hygiene cut idle RSS meaningfully (worker 390→251 MiB, scheduler 159→137, api 323→291, template process 206→111 MB), with no image split or deploy-shape change (#362).
- **Diagnostics memory now matches `kubectl top`** — reports cgroup working-set instead of `memory.current` (was ~2× over-reporting), which also corrects the admission-control 85% memory gate that fired prematurely (#279).
- **OAuth global-token re-stamp fix (with healing migration)** — a global integration connection had become unusable for every org except the one whose admin last refreshed it; ships an Alembic migration that heals mis-stamped tokens on deploy (#337).

## Developer Experience

- **CLI contract gate** — the CLI negotiates a `contract_version` with the server and hard-blocks on a mismatch, so an outdated CLI fails loudly instead of corrupting data.
- **Unified `--org` standard** — uniform org targeting across commands: `--org <id|name|none|global>` (with `--organization`/`--scope` as synonyms) plus a `--global` flag, with distinct home/global/org semantics.
- **`bifrost files` CLI** — `read / write / list / delete / exists / search` over the files SDK, plus `files.search()`, with the CLI shimmed into the worker image so workflows can shell out to it (#253).
- **`bifrost workflows remap`** — move references from one workflow ID to another active workflow (for replacing/renaming referenced workflows), plus a read-after-write guard so `agents update --tool-ids` fails loudly if the tool set doesn't persist (#353).
- **Windows onboarding** — native-PowerShell `Initialize-Bifrost.ps1` (no openssl/sed/Git Bash), RabbitMQ runs as `user:rabbitmq` to fix `.erlang.cookie` EACCES on Docker ≥24, UTF-8 CLI stdio (ends cp1252 crashes in push/watch), and POSIX repo keys (Windows backslash keys re-uploaded every file each watch tick) (#349).
- **SDK timeouts plumbed through** — `agents.run(timeout=)` no longer tears down at a hardcoded 30s; `knowledge.store_many(timeout=)` for large batch embeds (#302).
- **Self-host reference stack switched MinIO → SeaweedFS** — the bundled Compose/dev/debug/test examples ship SeaweedFS (different endpoints/credentials, no MinIO bucket-init); the app contract stays S3-compatible (#229 by @MTG-Thomas).
- WebAuthn passkey-transport normalization (#357 by @MTG-Thomas).

## Notable Fixes

- **Git-sync / manifest config import** — imported integration configs were landing with `config_schema_id = NULL` (unlinked, breaking resolution); now matched on `(integration_id, key)` with existing secret values preserved (#333).
- **Knowledge store chunks long documents on write and dedupes search** by `(namespace, organization_id, key)` so one long document can't dominate results — a behavior change for RAG users; reindexing re-chunks legacy rows (#285).
- **Table policy boolean/numeric comparisons** — `{eq: [{row: shared}, true]}` and numeric/ordering comparisons against JSONB fields previously raised "operator does not exist: text = boolean"; now correct (#268).
- **Documents PK promoted to composite `(table_id, id)`** — doc ids were globally unique instead of per-table, causing prod 404-read/409-write failures when two tables reused an id; schema migration runs on upgrade (#257).
- **Published apps no longer react to draft hot-reload** — draft bundle-rebuild events were bleeding into live apps; now gated behind preview mode (#338). App-relative location paths fixed for published apps (#351).
- **OAuth authorize-URL construction** — uses `&` when the template already has a query string (#288 by @sdc53).
- **Non-admin users no longer see workflow rerun/retry controls or workflow filtering** in execution surfaces (least-privilege behavior change) (#327).

## Breaking Changes

- **Removed `bifrost.email.send()`, the Settings → Email page, and `POST /api/email/send`.** This was an anti-pattern: it wired a fixed platform feature (sending mail) into arbitrary workflows through an ill-defined contract, and it's not a shape we want to repeat. The platform now exposes capabilities through **events** instead. **Migration:** if you have an email workflow today, adapt it to subscribe to a topic event (`await events.emit(topic, data)`) — you can reuse the body of your existing workflow as the subscriber. (#273)
- **Removed `bifrost export` / `bifrost import`.** Superseded by Solutions. Old invocations now error (exit 1) — no silent no-op. **Migration:** package with `bifrost solution deploy` / `solution export` and distribute with `bifrost solution install <zip>`; for raw `_repo/` workspace movement, use git sync.
- **App creation defaults to the v2 app model.** A bare `bifrost apps create <name>` (or `POST /api/applications` with no `app_model`) now returns **409**, directing you to scaffold/deploy v2 apps inside a Solution (`bifrost solution scaffold-app` + `solution deploy`). **Migration:** pass `--app-model inline_v1` (CLI) / `"app_model": "inline_v1"` (REST) to create a legacy standalone app.
- **CLI contract version advanced.** Older CLIs are blocked against the new server. **Migration:** upgrade your `bifrost` CLI to this release's build from `<API_URL>/api/cli/download`.
- **`access_level="authenticated"` now means "everyone except external users."** Inert on upgrade (all existing users default to non-external). **Migration:** when you start marking users external, move any entity they must reach to the new `everyone` tier or grant them an explicit role (see the External users feature above).
- **Deprecated the developer-context org-override mechanism** (part of removing that feature); pass `scope` explicitly per SDK call. Per-user override rows are dropped on upgrade.
- **Removed the obsolete CLI local-sessions UI/routes** (#350).
- Custom claims can now be global (`organization_id` widened to nullable) — additive, no action required.

### Migrations that run on upgrade

Several schema migrations apply automatically: composite documents PK `(table_id, id)` (#257), custom-claims org nullability (#291), the `everyone` access enum, OAuth token re-stamp healing (#337), and the Solutions tables. All are forward-safe on a populated DB.

## Solutions v1.0.0 — coverage caveats

Knowledge/RAG corpora, form/table file uploads, and private-repo install are **not** bundled/supported yet. Agents reference knowledge by namespace name, and the namespace must already exist in the target environment.
